## 0. Транспортная основа (что уже есть)

Прежде чем описывать изменения, необходимо понять базу, поверх которой строится mesh.

### Текущий стек

| Уровень | Технология | Детали |
|---------|-----------|--------|
| **Обнаружение** | mDNS (grandcat/zeroconf) + UDP broadcast 47821 | Оба механизма рекламируют кортеж `(peerID, name, tcpPort)` |
| **Транспорт** | TCP | Persistent соединения, один socket на пару узлов |
| **Framing** | Length-prefixed JSON | 4 байта big-endian uint32 длина + JSON тело |
| **Шифрование** | X25519 ECDH при handshake, ChaCha20-Poly1305 per-message | Shared key уникален для каждой пары пиров |
| **Топология** | Flat P2P | Каждый узел соединён только с теми, кого видит напрямую |

### Критическое ограничение текущей архитектуры

Сейчас `connByPeerID` хранит только **прямые** TCP-соединения. Если узел A не видит узел C напрямую (другая подсеть, слабый сигнал, стена), то A и C не могут общаться вообще. Задача mesh — позволить A отправлять сообщения C **через** B, который видит обоих.

### Почему TCP остаётся транспортом для relay

- Relay — это пересылка уже принятых сообщений по **существующим** TCP-соединениям. Нового транспортного протокола не нужно.
- UDP мог бы быть быстрее, но требует реализации надёжности поверх него. TCP уже обеспечивает ordered delivery.
- Сигналы звонков (WebRTC ICE/SDP) тоже идут через этот же канал — relay поможет пробросить их через ретранслятор.

## 1. Концепция: Flood Routing с TTL

### Почему Flood, а не Distance-Vector или Source Routing

**Distance-Vector (RIP-стиль):**
- Узлы периодически обмениваются таблицами маршрутизации
- Требует конвергенции при изменении топологии (~30с для RIP)
- Count-to-infinity проблема, split horizon
- Реализация: ~1000 строк, 2–3 недели

**Source Routing (узел сам прокладывает путь):**
- Отправитель должен знать полную топологию
- Нужен протокол обнаружения топологии (RREQ/RREP как в AODV)
- Хорошо для больших стабильных сетей
- Реализация: ~800 строк, 1–2 недели

**Flood Routing с TTL (выбранный подход):**
- Сообщение рассылается всем соседям; каждый пересылает дальше пока TTL > 0
- Дедупликация по msg.ID предотвращает петли и лишнюю нагрузку
- При TTL=5 и диаметре сети 3 хопа — сообщение дойдёт до всех
- Реализация: ~200–300 строк
- **Точно соответствует требованию ТЗ**: "продемонстрировать мультихоп-цепочку (пересылку через узел-ретранслятор)"

### Топология демонстрации

```
[A] ──TCP── [B] ──TCP── [C]
             ↑
         relay node
```

- A и C не видят друг друга напрямую
- B подключён к обоим
- A отправляет сообщение → B пересылает → C получает
- HopPath в сообщении: `[A → B → C]`

## 2. Изменения в протоколе (`go/core/message.go`)

### 2.1 Новые поля в структуре Message

Необходимо добавить три поля в существующую структуру `Message`:

**TTL (Time To Live)**
- Тип: `int`
- JSON-ключ: `"ttl,omitempty"`
- Назначение: количество оставшихся хопов. Каждый ретранслятор уменьшает на 1 перед пересылкой. Если TTL ≤ 0 — пересылка запрещена.
- Значение по умолчанию при создании: 5 (достаточно для сети диаметром до 5 хопов)
- Почему 5: при большем TTL растёт нагрузка на сеть. 5 хопов — разумный максимум для LAN-mesh.

**Origin**
- Тип: `string` (peerID)
- JSON-ключ: `"origin,omitempty"`
- Назначение: peerID исходного отправителя. Не изменяется ретрансляторами. Нужен для:
  - построения routing table (Origin доступен через viaConn)
  - корректного определения отправителя при relay (msg.PeerID может быть ретранслятором)
  - дедупликации (в связке с msg.ID)

**HopPath**
- Тип: `[]string` (slice of peerID)
- JSON-ключ: `"hop_path,omitempty"`
- Назначение: список peerID всех узлов, через которые прошло сообщение. Каждый ретранслятор добавляет свой ID. Используется для:
  - отображения пути в UI ("через B")
  - диагностики и логирования
  - обнаружения топологии (кто с кем соединён)
- Примечание: HopPath не влияет на логику маршрутизации, только на наблюдаемость.

### 2.2 Новый тип сообщения: MsgTypeRelay

Добавить константу `MsgTypeRelay MsgType = "relay"`.

Relay-сообщение — это специальная обёртка для случаев, когда пересылается зашифрованное личное сообщение (Direct, GroupText, FileChunk и т.д.). Ретранслятор не может расшифровать содержимое (нет ключа), поэтому пересылает как есть.

**Альтернатива**: можно не вводить новый тип, а просто форвардить исходное сообщение с TTL>0. Выбрать этот подход — проще и достаточно.

### 2.3 Изменения в существующих конструкторах

При вызове `NewTextMessage`, `NewDirectMessage` и других конструкторов:
- Установить `TTL = 5`
- Установить `Origin = n.id` (peerID отправителя)
- `HopPath` остаётся пустым (заполняется ретрансляторами)

## 3. Дедупликация (`go/core/node.go`)

Дедупликация — **фундамент** mesh-сети. Без неё flood создаёт бесконечные циклы.

### 3.1 Хранилище seen-сообщений

Добавить в структуру `Node`:

```
seenMsgIDs: отображение msgID -> время получения
seenMsgMu:  мьютекс (или использовать sync.Map)
```

**Почему sync.Map**: конкурентные операции LoadOrStore являются атомарными, не требуют явного мьютекса.

### 3.2 Алгоритм дедупликации

В начале `readLoop`, **до** switch по типу сообщения, для каждого входящего сообщения:

1. Попытаться атомарно записать `msgID → time.Now()` в seenMsgIDs
2. Если запись не удалась (ключ уже существует) — сообщение уже видели, **пропустить без обработки**
3. Если запись удалась — обрабатывать и пересылать как обычно

**Важно**: дедупликация должна происходить до relay-пересылки. Иначе: A→B→C→B (петля) — B получит своё же сообщение обратно от C и перешлёт снова.

### 3.3 Очистка кеша (TTL-based eviction)

Seen-кеш нужно периодически очищать, иначе он растёт бесконечно.

Запустить фоновую горутину `cleanSeenCache()`:
- Каждые 5 минут проходить по всем записям
- Удалять записи старше 30 минут
- 30 минут — достаточно, учитывая что TTL сообщений ограничен 5 хопами и временем обхода

Альтернатива — LRU-кеш с фиксированным размером (например, 10 000 записей). Но простая очистка по времени проще и достаточно надёжна.

## 4. Таблица маршрутизации (`go/core/node.go`)

Routing table — оптимизация поверх flood routing. Без неё всё работает (flood), но с ней — эффективнее.

### 4.1 Структура таблицы

Добавить в `Node`:

```
routeTable: map[targetPeerID] → nextHopPeerID
routeMu:    sync.RWMutex
```

Семантика: "чтобы достучаться до targetPeerID, нужно отправить сообщение через nextHopPeerID".

**Пример**:
- A подключён к B напрямую → `routeTable["B"] = "B"` (nextHop = сам peer)
- A получил сообщение от C через B → `routeTable["C"] = "B"` (C доступен через B)

### 4.2 Заполнение таблицы

Таблица заполняется **пассивно** — при обработке входящих сообщений.

При получении любого сообщения через соединение `cn` (peerID = `cn.peerID`):
- Если `msg.Origin` ≠ "" и `msg.Origin` ≠ `n.id`:
  - Записать `routeTable[msg.Origin] = cn.peerID` (Origin доступен через прямого соседа cn)
- Дополнительно — разбирать HopPath:
  - Последний элемент HopPath перед нами — тоже доступен через cn
  - Промежуточные узлы в HopPath недоступны напрямую, но косвенно

**Почему пассивное заполнение, а не активный exchange:**
- Активный обмен таблицами (как в DV-routing) требует отдельного протокола и периодических сообщений
- Пассивное заполнение работает автоматически — как только C отправит любое сообщение, A узнает путь до C
- Достаточно для демо

### 4.3 Использование таблицы при отправке

При вызове `sendDirect(text, toPeerID)`:
1. Проверить наличие прямого соединения `connByPeerID[toPeerID]`
2. Если прямое соединение есть — использовать его (текущее поведение)
3. Если прямого нет — поискать в `routeTable[toPeerID]`
4. Если найден nextHop → отправить сообщение через `connByPeerID[nextHop]` с установленным `To = toPeerID` и `TTL > 0`
5. Если не найден → flood всем соседям с `TTL = 5`

## 5. Логика пересылки (relay forwarding) (`go/core/node.go`)

### 5.1 Когда форвардить

Условия для форвардинга входящего сообщения:

1. `msg.TTL > 0` — TTL не исчерпан
2. Сообщение **не предназначено нам** (`msg.To != n.id` для unicast) ИЛИ это broadcast (`msg.To == ""`)
3. Дедупликация пропустила (сообщение новое)

**Типы сообщений, которые нужно форвардить:**
- `MsgTypeText` (broadcast) — форвардить всем соседям кроме источника
- `MsgTypeDirect` — если `msg.To != n.id`, форвардить к nextHop или флудить
- `MsgTypeGroupText` — форвардить участникам группы, которые недоступны напрямую
- `MsgTypeFileOffer`, `MsgTypeFileChunk`, `MsgTypeFileDone` — форвардить к получателю
- `MsgTypeCallOffer`, `MsgTypeCallAnswer`, `MsgTypeCallIce` и прочие call-сигналы — форвардить

**Типы сообщений, которые НЕ форвардить:**
- `MsgTypeHello` — handshake только между прямыми соседями
- `MsgTypePing` / `MsgTypePong` — keepalive только для прямого соединения
- `MsgTypeLeave` — управление прямым соединением

### 5.2 Алгоритм функции `maybeForward(msg, sourcePeerID)`

```
Вход: msg (Message), sourcePeerID (peerID соседа, от которого получили)

1. Если msg.TTL ≤ 0: вернуть (TTL исчерпан)

2. Уменьшить msg.TTL на 1

3. Добавить n.id в конец msg.HopPath

4. Определить список получателей:
   a. Если msg.To == "" (broadcast):
      - Получатели = все прямые соединения, кроме sourcePeerID
   b. Если msg.To != "" (unicast):
      - Поискать nextHop в routeTable[msg.To]
      - Если найден и nextHop != sourcePeerID:
        - Получатели = [nextHop]
      - Иначе:
        - Получатели = все прямые соединения, кроме sourcePeerID (flood)

5. Отправить msg каждому из получателей через cn.send(msg)

6. Логировать: [relay] msg_id=X from=sourcePeerID to=[получатели] TTL=N path=HopPath
```

### 5.3 Где вызывать maybeForward

В `readLoop` после блока switch, для обрабатываемых типов сообщений:

- Для `MsgTypeText`: **после** emit события — вызвать `maybeForward`
- Для `MsgTypeDirect` где `msg.To != n.id`: вместо `continue` — вызвать `maybeForward`
- Для call-сигналов где `msg.To != n.id`: вместо `continue` — вызвать `maybeForward`
- Для `MsgTypeFileOffer/Chunk/Done` где получатель не мы: вызвать `maybeForward`

**Ключевой момент**: для `MsgTypeDirect` нужно проверить `msg.To` **до** попытки расшифровки. Если `msg.To != n.id` — мы не можем расшифровать (нет ключа с оригинальным отправителем), просто форвардим зашифрованный payload как есть.

### 5.4 Функция `floodExcept(msg, excludePeerID)`

Вспомогательная функция для рассылки сообщения всем соседям кроме одного:

```
1. Взять snapshot connByPeerID под Lock
2. Освободить Lock
3. Для каждого (peerID, cn) кроме excludePeerID:
   - cn.send(msg)
4. Ошибки send логировать, не прерывать loop
```

**Почему snapshot**: чтобы не держать Lock во время IO. Отправка в TCP может блокироваться.

## 6. Интеграция с Discovery (обнаружение через relay)

### 6.1 Проблема

Сейчас `Discovery` обнаруживает только прямых соседей через mDNS/UDP-broadcast. Узлы за ретранслятором не видны через Discovery.

### 6.2 Peer Announcement через Relay

Ввести новый тип сообщения: `MsgTypePeerAnnounce`

Когда узел B подключается к новому узлу C:
1. B рассылает всем своим соседям (включая A) сообщение типа `MsgTypePeerAnnounce` с полями:
   - `PeerID` = C.id
   - `Name` = C.name
   - Нет прямого адреса (C недоступен напрямую)
   - TTL = 3 (не нужно далеко)

Когда A получает MsgTypePeerAnnounce:
- A добавляет C в `PeerManager` со статусом `PeerReachableViaRelay`
- A обновляет `routeTable["C"] = B`
- A **не** пытается TCP-dial к C (нет прямого адреса)
- A эмитит событие `PeerJoined` (Flutter UI показывает нового пира в списке)

### 6.3 Маршрутизация к relay-пирам

При `sendDirect(text, "C")`:
- `connByPeerID["C"]` = nil (нет прямого соединения)
- `routeTable["C"]` = "B"
- Отправить через `connByPeerID["B"]` с `To = "C"`, `TTL = 5`

### 6.4 Шифрование для relay-сообщений

**Проблема**: для Direct-сообщений требуется shared key с получателем. Но если A никогда не соединялся с C напрямую, у A нет ECDH-ключа для C.

**Решение**: при получении MsgTypePeerAnnounce, A также получает публичный ключ C (`PubKey`). A выполняет ECDH со своим private key и pubkey C — получает shared key. Это стандартный ECDH без прямого соединения.

Для этого добавить в MsgTypePeerAnnounce поле `PubKey []byte`.

**Безопасность**: ретранслятор B видит зашифрованный payload, но не может его расшифровать (нет ключа A-C). End-to-end encryption сохраняется.

### 10.2 Unit-тест: relay 3-узловая цепочка

Файл: `go/core/node_test.go`

Создать три узла A, B, C на localhost:0. Соединить только A↔B и B↔C (A и C не видят друг друга). Отправить текстовое сообщение от A. Убедиться что:

- B **не** эмитит это сообщение как входящее (только пересылает)

### 10.3 Unit-тест: TTL

Отправить сообщение с TTL=1 через цепочку A→B→C→D. Убедиться что D не получает сообщение (TTL=0 у C, C не форвардит к D).

### 10.4 Unit-тест: защита от петли

Соединить A↔B и B↔A (дублирующее обратное соединение для теста). Отправить сообщение. Убедиться что нет бесконечного цикла — сообщение обрабатывается ровно один раз каждым узлом.

### 10.5 Integration-тест: relay-отключение

Тест отказоустойчивости:
1. Создать цепочку A↔B↔C
2. Отправить сообщение от A к C (приходит через B)
3. Остановить B (симулировать отключение ретранслятора)
4. Подключить A напрямую к C
5. Убедиться что A снова может общаться с C напрямую (routing table обновляется)

## 11. Потенциальные проблемы и решения

### P-1. Flood нагружает сеть при большом числе узлов

**Проблема**: при 10+ узлах каждое broadcast-сообщение рождает O(n²) копий.

**Решение**: TTL=3 вместо 5 для broadcast (text) сообщений. Для unicast (direct, call) — TTL=5, так как нужна гарантия доставки. Дедупликация уже устраняет дубли на каждом узле.

**Для демо**: 3–5 узлов, нагрузка незначительна.

### P-2. HopPath растёт и раздувает сообщения

**Проблема**: при TTL=5 HopPath может содержать до 5 UUID (5 × 36 байт = 180 байт overhead).

**Решение**: допустимо для демо. В продакшне — truncate HopPath до последних 3 хопов.

### P-3. Routing table устаревает

**Проблема**: узел B уходит, но routeTable["C"] = "B" остаётся. Следующая отправка к C идёт через несуществующий B и падает.

**Решение**: при PeerLeft-событии (B ушёл) удалять из routeTable все записи где nextHop = B. Затем flood для обнаружения новых путей.

### P-4. ECDH без прямого соединения (pubkey доверие)

**Проблема**: когда A получает pubkey C через ретранслятора B, B мог подменить pubkey C (MITM).

**Решение для демо**: принять как known limitation, задокументировать в модели угроз. Полное решение — Ed25519 подпись pubkey identity key (Фаза 5 из TODO.md).

### P-5. Большие файлы через relay

**Проблема**: file chunks (~512 KB каждый) пересылаются через ретранслятора, удваивая нагрузку на B.

**Решение**: не форвардить `MsgTypeFileChunk` если `msg.To != n.id`. Файлы передаются только при прямом соединении. В FileOffer уведомить пользователя если получатель доступен только через relay — передача невозможна без прямого соединения. Это разумное ограничение для демо.

### P-6. WebRTC через relay (ICE signaling)

**Проблема**: call_offer, call_ice, call_answer идут через relay-канал. WebRTC сам устанавливает прямое медиа-соединение (peer-to-peer), но ICE candidates содержат IP-адреса. Если A и C в разных подсетях, ICE может не найти прямой путь.

**Решение**: call signaling форвардить через relay. WebRTC попытается установить соединение самостоятельно. Если ICE не сможет — звонок не состоится (нет STUN/TURN). Для демо: A и C должны быть в одной LAN (просто без прямого видения друг друга на уровне приложения). В этом случае ICE найдёт путь по LAN.

## 13. Что НЕ входит в план

| Исключение | Причина |
|-----------|---------|
| Distance-vector / OSPF routing | Сложность несоразмерна демо. Flood достаточен для ТЗ |
| AODV (reactive routing) | Требует RREQ/RREP протокол обнаружения маршрутов — ~500 строк дополнительно |
| File transfer через relay | Нецелесообразно для больших файлов (удвоение трафика на ретрансляторе) |
| BLE/WiFi Direct как транспорт для relay | Другой транспортный уровень — отдельная задача |
| NAT traversal (STUN/TURN) | Требует внешнего сервера — противоречит децентрализации |
| Multi-path routing | Слишком сложно, избыточно для демо |
